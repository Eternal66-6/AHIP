⚠️ The Problem Before This Stage

Agents ran independently in parallel. If the Patient Agent found a severe "High Risk" clinical issue, the Claims Agent didn't know about it, resulting in the claim being treated as standard priority. They lacked teamwork.

🛠️ What We Built

🧠 SharedCaseMemory Schema

A central state object containing an array of flags (e.g. `HIGH_RISK_PATIENT`) that gets passed between agents during execution.

⛓️ Sequential Pipeline Orchestrator

Refactored the orchestrator to run agents in a specific order so downstream agents can read upstream memory.

👨‍⚖️ Consolidator Agent

A final "manager" agent that reads all the outputs and flags, resolving conflicts and generating one single definitive routing output.

🔄 Before vs After

Before Phase 4

  • Parallel, isolated agent execution
  • Downstream agents blind to upstream findings
  • Multiple conflicting outputs generated

After Phase 4

  • Sequential, stateful orchestration
  • Shared Memory passes `flags` between agents

🎙️ Interview Q&A

Q: How do your agents communicate with each other?
A: Through a structured `SharedCaseMemory` object. We use a sequential pipeline where Agent A writes flags to the memory, and Agent B reads that memory before running its logic, allowing cross-domain collaboration.
Q: Why did you add a Consolidator Agent?
A: When multiple agents generate outputs, human operators get confused. The Consolidator Agent acts as a manager; it synthesizes all agent findings into one definitive `ConsolidatedCaseOutput` indicating the final routing destination.
