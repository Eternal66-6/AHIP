# AHIP Product Walkthrough

This document maps the user-facing UI features of the AHIP platform back to the underlying technical Agentic architecture.

## 1. Executive Dashboard (Data Layer)
**What the user sees:** Tables of Patients, Claims, and Providers, alongside high-level metrics.
**What happens backend:** A standard FastAPI REST layer built over a PostgreSQL database using SQLAlchemy ORM. This represents the "legacy" data state of an enterprise before AI is applied.

## 2. Priority Queue (Decision & Escalation Layer)
**What the user sees:** A sorted list of claims, ranked by severity, with red flags for high-risk items and clear Explainability Notes.
**What happens backend:** The frontend hits the `GET /priority-queue` endpoint (built in Phase 5). This endpoint queries the `AgentExecutionLog` for the final output of the `Consolidator Agent`. It applies a deterministic sorting algorithm (High Risk cases bubble to the top) over the unstructured JSON outputs generated by the LLM.

## 3. Case Details (Context & Collaboration Layer)
**What the user sees:** A side-by-side view of Knowledge Graphs (Context Packs) and Agent Execution Logs.
**What happens backend:** 
- **Context Engine (Phase 3):** When a case is queried, the `HealthcareContextBuilder` extracts raw database rows (Patient, Provider, Claim) and transforms them into heavily minimized, Pydantic-validated "Context Packs".
- **Multi-Agent Collaboration (Phase 4):** When "Run Agents" is clicked, a multi-stage pipeline executes. The Claims Agent analyzes the claim context, the Compliance Agent analyzes the provider context, and their outputs are merged into a `SharedCaseMemory` object. The Consolidator Agent reads this shared memory and outputs the final JSON recommendation.

## 4. Governance & Audit (Security Layer)
**What the user sees:** An immutable table tracking who clicked what, and what the AI did.
**What happens backend:** (Built in Phase 6). The `POST /decision` endpoint requires a human justification. This action is permanently written to a new `AuditLog` Postgres table. The frontend uses a simulated Role-Based Access Control (RBAC) token to hide this page from non-compliance staff.
