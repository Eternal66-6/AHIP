⚠️ The Problem Before This Stage

In Phase 4, our AI agents successfully collaborated and wrote incredibly detailed findings to our `AgentExecutionLog` database table. But humans don't read database tables.

Operations teams were left looking at raw JSON logs. We had no way to rank which cases were SLA-risks, no way for humans to intervene, and no clean UI to prove why the AI made a decision.

🛠️ What We Built

⚖️ Risk Scoring Model

A backend heuristic in `agents.py` that evaluates case flags and automatically elevates "High Risk" cases to the top of the queue.

🔍 Priority Queue API

The `GET /priority-queue` endpoint that fetches consolidated outputs and deduplicates them by `case_id`.

🧠 Explainability UI

A dedicated dashboard in React (`App.tsx`) showing the AI's step-by-step reasoning natively beside every routing decision.

👤 Human-in-the-Loop

The `POST /decision` API allowing human operators to "Accept" or "Override" the AI recommendation directly.

💻 Implementation Highlights

From the Repomix context, here is how we deduplicated and scored the priority queue in Python:

@router.get("/priority-queue")
def get_priority_queue(db: Session = Depends(get_db)):
    logs = db.query(AgentExecutionLog).filter(AgentExecutionLog.agent_name == "Consolidator Agent").all()
    
    # Deduplicate by case_id
    seen_cases = set()
    unique_logs = []
    for log in logs:
        if log.case_id not in seen_cases:
            seen_cases.add(log.case_id)
            unique_logs.append(log)
    
    # Risk Scoring Model: High risk first, then by date
    def sort_key(log):
        is_high_risk = "Senior Claims Analyst" in log.related_workflow_event or "High" in log.observation
        return (0 if is_high_risk else 1, log.created_at)
        
    return sorted(unique_logs, key=sort_key)

🎙️ Interview Q&A

Q: How do you handle AI hallucinations in your architecture?
A: By designing a Human-in-the-Loop (HITL) Priority Queue. The AI generates a recommendation and an Explainability Note, but it sits in a queue until an Operations Analyst clicks 'Accept' or 'Override'.
Q: How does your system prioritize which cases a human reviews first?
A: We built a Risk Scoring Model via our `GET /priority-queue` API. It parses the ConsolidatorAgent's outputs and applies a sorting heuristic, forcing High Risk SLA cases to index 0.
Q: What happens if an agent is run multiple times on the same case?
A: I implemented a backend deduplication step using a Python `set()` on the `case_id` to ensure the priority queue only renders the single most recent decision for any given case, preventing UI bloat.
